field guide · state management

Where does this state live?

Every piece of state answers one question: who else needs to see it, and how fresh does it have to stay? Answer that, and the right tool follows. This guide walks through the four you'll meet most often — local state, Context, Redux, and React Query.

quick decision helper — answer three questions

Does this data come from a server — an API, a database, anything fetched over the network?

Does more than one distant component need to read it?

Does it change often, from many places, or need time-travel debugging and middleware?

the four systems

LOCAL STATE

useState

State that one component owns and no one else needs. The default until proven otherwise.

const [count, setCount] = useState(0);

reach for it when

  • Only this component and its children care
  • It resets fine on unmount
  • Form fields, toggles, hover state, scratch input

skip it when

  • Two unrelated components need to stay in sync
CONTEXT API

Context

A way to pass a value down the tree without threading props through every level in between.

const Theme = createContext(); // <Theme.Provider value={...}>

reach for it when

  • Theme, locale, auth session, feature flags
  • Reads are far more common than writes
  • You want plain React, no extra library

skip it when

  • Value updates rapidly — every consumer re-renders
REDUX

Redux

A single store for client state, updated only through dispatched actions — predictable, traceable, inspectable.

dispatch({ type: 'task/added' });

reach for it when

  • State is large, shared, and updated from many places
  • You want action history, time travel, middleware
  • Several teams touch the same slice of state

skip it when

  • The data actually comes from a server — that's a cache problem, not a store problem
REACT QUERY

React Query

Not state management — cache management. It tracks server data: fetching, retries, staleness, refetching.

const { data } = useQuery(['stats'], fetchStats);

reach for it when

  • Data is fetched, and can go stale
  • You want loading/error states for free
  • Polling, pagination, background refetch

skip it when

  • The value never touches a network — plain state or Context does it with less overhead
